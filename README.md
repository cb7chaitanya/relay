# Relay — AI-Powered Customer Support Chat

A real-time customer support chat application with streaming AI responses, built as a monorepo with a Next.js frontend and Express backend.

## Architecture

```
apps/
  web/    → Next.js 15, React 19, TailwindCSS    (Vercel)
  api/    → Express 4, Prisma, TypeScript          (Render)
packages/
  shared/ → Zod schemas, shared types              (internal)
```

### Backend Layers

```
Routes → Services → Repositories → Prisma/PostgreSQL
                  → Providers    → LLM (OpenAI / OpenRouter)
```

- **Routes** — HTTP handling, SSE formatting, validation middleware
- **Services** — Business logic, orchestration. Provider-agnostic — never imports SDK types
- **Repositories** — Data access via Prisma. All queries are functions, not classes
- **Providers** — LLM abstraction. `LLMProvider` interface with `OpenAIProvider` and `OpenRouterProvider` implementations. Factory selects at startup based on `LLM_PROVIDER` env var

### Frontend Layers

- **`useChat` hook** — Single source of truth for all chat state (messages, loading, streaming, errors, session)
- **Components** — Presentational. Receive data and callbacks via props
- **`lib/api.ts`** — Fetch wrapper with SSE stream parser

### Key Design Decisions

| Decision | Rationale |
|---|---|
| SSE streaming (not WebSocket) | One-directional server→client flow. Simpler than WebSocket, works over HTTP/1.1, no connection upgrade needed |
| Optimistic user messages | Message appears instantly in the UI before the backend confirms. Reduces perceived latency |
| Provider abstraction | `LLMProvider` interface decouples the service layer from any specific LLM SDK. Swap providers via env var |
| `sessionId` in localStorage | Simplest anonymous session persistence. No auth system needed for the chat use case |
| No shared code beyond types | `packages/shared` only contains Zod schemas and TypeScript interfaces. No runtime logic shared — keeps deploy units independent |
| Server-side UUIDs | All IDs generated by the database/server, never the client. Prevents ID collisions and spoofing |
| 10-message context window | Limits token usage per request while maintaining conversation coherence. Configurable constant |

### Tradeoffs

- **No auth** — Anyone with a `sessionId` can read that conversation. Acceptable for anonymous support chat; would need auth for production with sensitive data
- **No message pagination** — All messages loaded at once on session restore. Fine for support conversations (typically <50 messages); would need cursor-based pagination for longer threads
- **In-memory rate limiting** — Resets on server restart. Sufficient for single-instance deploys; use Redis for multi-instance

## Setup

### Prerequisites

- Node.js ≥ 20
- PostgreSQL
- An OpenRouter or OpenAI API key

### Install

```bash
npm install
```

### Configure

```bash
cp apps/api/.env.example apps/api/.env
# Edit apps/api/.env with your database URL and API key

cp apps/web/.env.example apps/web/.env.local
# Edit NEXT_PUBLIC_API_URL if needed
```

### Database

```bash
cd apps/api
npx prisma migrate dev
npx tsx prisma/seed.ts   # optional: seed sample data
```

### Run

```bash
# From project root
npm run dev:api    # Express on :3001
npm run dev:web    # Next.js on :3000
```

## API

| Method | Path | Description |
|---|---|---|
| `POST` | `/chat/message` | Send message, receive SSE stream |
| `GET` | `/chat/session/:id` | Restore conversation history |
| `DELETE` | `/chat/session/:id` | Delete conversation |
| `GET` | `/health` | Health check (DB + provider status) |

### SSE Events (POST /chat/message)

```
event: session     → { sessionId: string }
event: token       → { token: string }
event: done        → { promptTokens, completionTokens }
event: error       → { error: string, code: string }
```

### Rate Limiting

`POST /chat/message` is rate-limited to 20 requests per minute per IP.

## Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `PORT` | No | `3001` | API server port |
| `DATABASE_URL` | Yes | — | PostgreSQL connection string |
| `LLM_PROVIDER` | No | `openrouter` | `openai` or `openrouter` |
| `OPENAI_API_KEY` | If provider=openai | — | OpenAI API key |
| `OPENROUTER_API_KEY` | If provider=openrouter | — | OpenRouter API key |
| `CORS_ORIGIN` | No | `http://localhost:3000` | Allowed CORS origin |
| `NEXT_PUBLIC_API_URL` | No | `http://localhost:3001` | API URL for frontend |

## Deployment

### Backend (Render)

- Build: `npm run build:shared && npm run build -w @relay/api`
- Start: `npm run start -w @relay/api`
- Health check: `GET /health`

### Frontend (Vercel)

- Framework: Next.js
- Root directory: `apps/web`
- Build: handled by Vercel
- Set `NEXT_PUBLIC_API_URL` to the Render URL

### Database (Neon)

- Set `DATABASE_URL` in Render environment to the Neon connection string
- Run `npx prisma migrate deploy` as part of the build step
